zval是php数据类型和垃圾回收的基石，也是众多问题的根源所在。先了解一下zval是什么。

抛开其它的概念（稍后分析），先对zval有一个直观的认识：一个在PHP中可以动态存储多种c语言数据类型的结构。也可以暂时把一个zval理解为一个升级版的union变量。

3、基于引用计数的垃圾回收

最初，PHP的垃圾回收机制采用了引用计数（Reference Counting）的方法。

当我们在对一个PHP变量进行操作时，实际上发生的事情并没有我们想象的那么简单。

变量a，是在当前作用域中生成的。并且生成了类型为 string 和值为1g的zval容器。在额外的两个字节信息中，"is_ref"被默认设置为 false，因为没有任何自定义的引用生成。"refcount"被设定为 1，因为这里只有一个变量使用这个变量容器。当"refcount"的值是1时，"is_ref"的值总是false。

刚开始，引用次数是2，因为同一个变量容器被变量a 和变量b关联。在这种情况下，php不会去复制已生成的变量容器。当任何关联到某个变量容器的变量离开它的作用域(比如：函数执行结束)，或者对变量调用了函数 unset()时，”refcount“就会减1。变量容器在”refcount“变成0时就被销毁。

上述分析的是简单类型的过程，对于array、object等复合类型则更为复杂。比如说生成一个如下的array：$a=array('name' => '1g', 'age' => 19 )，debug会显示你同时创建了3个zval！

感兴趣的读者可以在PHP5中试一试各种玩法。

5、回收机制的改进

由于引用计数的垃圾回收机制无法处理循环引用导致的内存泄露问题，因此在PHP5.3之后，引入了基于同步回收周期的方法。

该算法的大致规则如下：

首先，把像上文提到的类似a数组这样的zval称为possible root，（因为这个zval可能附带了一堆zval，就像一颗树）。

A 填充：把所有的root，放在一个root buffer里面。这个缓冲区里的root，用紫色标记，表示疑似垃圾。仅仅在root缓冲区满了时，才对缓冲区内部所有不同的zval执行垃圾回收操作。

B 模拟删除：模拟删除每个紫色变量。模拟删除时可能将不是紫色的普通变量引用数减”1”，如果某个普通变量引用计数变成0了，就对这个普通变量再做一次模拟删除。每个变量只能被模拟删除一次，模拟删除后标记为灰。

C 模拟恢复：模拟恢复每个紫色变量。恢复是有条件的，当变量的引用计数大于0时才对其做模拟恢复。同样每个变量只能恢复一次，恢复后标记为黑，基本就是步骤 B 的逆运算。这样剩下的一堆没能恢复的就是应该删除的蓝色节点了，在步骤 D 中，把它们遍历出来真的删除掉。

D 真删除。

该算法的时间复杂度为O(N)，N为参与的节点数。

php.ini的zend.enable_gc定义了这个算法的设置。

其中主要的就是gc_mark_roots、gc_scan_roots和gc_collect_roots这三个函数

  • gc_mark_roots对gc_root_buffer中的每个节点调用zval_mark_greyzval_mark_grey；对所有节点的refcount减1并标记为灰色；对已标记为灰色的节点不处理。
  • gc_scan_roots调用zval_scan对每个节点进行处理，zval_scan只处理灰色节点；调用zval_scan_black对节点refcount大于0的节点的refcount加1并标记为黑色。refcount为0的节点标记为白色。
  • gc_collect_roots把所有白色节点放入gc_free_list链表等待释放。
